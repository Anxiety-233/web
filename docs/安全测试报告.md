## 测试一：JWT Token 伪造/篡改攻击

### 攻击目的
验证系统能否识别伪造或篡改的 JWT Token

### 攻击步骤

**1. 先正常登录获取合法 Token：**
```
curl.exe -X POST http://hn.frp.one:38654/api/auth/login -H "Content-Type: application/json" -d "{\"username\":\"admin\",\"password\":\"123456\"}"
```

**2. 使用伪造的 Token 访问受保护接口：**
```
curl.exe -X GET http://hn.frp.one:38654/api/equipment/list -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.FAKE_TOKEN.invalid_signature" -H "timestamp: 1716300000000" -H "nonce: test-fake-token-001"
```

### 预期结果
返回 HTTP 401，响应体包含 "Token无效" 或 "Token签名验证失败"

### 防御机制
- JwtAuthenticationFilter 使用 HS256 + 服务端密钥验证签名
- 签名不匹配时抛出 SignatureException，返回 401

## 测试二：Token 过期验证

### 攻击目的
验证过期 Token 无法继续使用

### 攻击步骤

**使用一个已过期的 Token（手动构造或等待 15 分钟后重试）：**
```
curl.exe -X GET http://hn.frp.one:38654/api/equipment/list -H "Authorization: Bearer <过期的token>" -H "timestamp: 1716300000000" -H "nonce: test-expired-001"
```

### 预期结果
返回 HTTP 401，提示 "Token已过期"

### 防御机制
- Access Token 有效期 15 分钟（900000ms）
- JwtUtil.isTokenExpired() 检查 exp 字段

## 测试三：重放攻击防御

### 攻击目的
验证相同请求不能被重复提交（防重放）

### 攻击步骤

**1. 先正常登录获取 Token：**
```
curl.exe -X POST http://hn.frp.one:38654/api/auth/login -H "Content-Type: application/json" -d "{\"username\":\"admin\",\"password\":\"123456\"}"
```

**2. 发送一个正常请求（记录 nonce 值）：**
```
curl.exe -X GET http://hn.frp.one:38654/api/equipment/list -H "Authorization: Bearer <你的token>" -H "timestamp: <当前毫秒时间戳>" -H "nonce: replay-test-unique-id-001"
```

**3. 立即重放完全相同的请求（相同 nonce）：**
```
curl.exe -X GET http://hn.frp.one:38654/api/equipment/list -H "Authorization: Bearer <你的token>" -H "timestamp: <当前毫秒时间戳>" -H "nonce: replay-test-unique-id-001"
```

### 预期结果
- 第一次请求：200 成功
- 第二次请求：400，响应 "请求重复（nonce已使用）"

### 防御机制
- AntiReplayInterceptor 将 nonce 存入 Redis（SETNX）
- 5 分钟内相同 nonce 的请求被拒绝
- timestamp 超过 5 分钟的请求直接拒绝

## 测试四：时间戳过期攻击

### 攻击目的
验证过期时间戳的请求被拒绝

### 攻击步骤

**使用一个 10 分钟前的时间戳发送请求：**
```
curl.exe -X GET http://hn.frp.one:38654/api/equipment/list -H "Authorization: Bearer <你的token>" -H "timestamp: 1000000000000" -H "nonce: old-timestamp-test-001"
```

### 预期结果
返回 HTTP 400，响应 "请求已过期（5分钟内有效）"

### 防御机制
- AntiReplayInterceptor 校验 |当前时间 - 请求时间戳| <= 300000ms
- 超过 5 分钟窗口的请求直接拒绝

## 测试六：未授权访问（无 Token）

### 攻击目的
验证未登录用户无法访问受保护资源

### 攻击步骤

**不带任何 Token 直接访问设备接口：**
```
curl.exe -X GET http://hn.frp.one:38654/api/equipment/list -H "timestamp: <当前毫秒时间戳>" -H "nonce: no-auth-test-001"
```

### 预期结果

### 防御机制
- UnauthorizedInterceptor 检查 SecurityContext 中的认证信息
- 匿名用户（anonymousUser）被拦截

## 测试七：水平越权攻击

### 攻击目的
验证普通用户不能操作他人设备

### 攻击步骤

**1. 用普通用户 user01 登录：**
```
curl.exe -X POST http://hn.frp.one:38654/api/auth/login -H "Content-Type: application/json" -d "{\"username\":\"user01\",\"password\":\"123456\"}"
```

**2. 尝试操作属于 admin（user_id=1）的设备：**
```
curl.exe -X POST "http://hn.frp.one:38654/api/equipment/update?id=1" -H "Authorization: Bearer <user01的token>" -H "Content-Type: application/json" -H "timestamp: <当前毫秒时间戳>" -H "nonce: privilege-test-001" -d "{\"status\":0}"
```

### 预期结果
返回 HTTP 403，响应 "无权操作他人设备"

### 防御机制
- HorizontalPrivilegeInterceptor 校验设备 user_id 与当前登录用户 ID 是否匹配
- 管理员（admin）直接放行，普通用户只能操作自己的设备

## 测试八：Token 黑名单（登出后 Token 失效）

### 攻击目的
验证用户登出后，旧 Token 立即失效

### 攻击步骤

**1. 登录获取 Token：**
```
curl.exe -X POST http://hn.frp.one:38654/api/auth/login -H "Content-Type: application/json" -d "{\"username\":\"admin\",\"password\":\"123456\"}"
```

**2. 使用该 Token 登出：**
```
curl.exe -X POST http://hn.frp.one:38654/api/auth/logout -H "Authorization: Bearer <你的token>" -H "timestamp: <当前毫秒时间戳>" -H "nonce: logout-test-001"
```

**3. 再次使用同一个 Token 访问接口：**
```
curl.exe -X GET http://hn.frp.one:38654/api/equipment/list -H "Authorization: Bearer <已登出的token>" -H "timestamp: <当前毫秒时间戳>" -H "nonce: after-logout-test-001"
```

### 预期结果
- 登出成功：200
- 再次使用：401，Token 已在黑名单中

### 防御机制
- 登出时 JwtUtil.addToBlacklist() 将 Token 存入 Redis 黑名单
- 黑名单过期时间 = Token 剩余有效期
- validateToken() 每次都检查黑名单

## 测试九：SQL 注入攻击

### 攻击目的
验证登录接口能否防御 SQL 注入

### 攻击步骤

**尝试 SQL 注入登录：**
```
curl.exe -X POST http://hn.frp.one:38654/api/auth/login -H "Content-Type: application/json" -d "{\"username\":\"admin' OR '1'='1\",\"password\":\"anything\"}"
```

### 预期结果
返回认证失败（401 或错误提示），不会绕过登录

### 防御机制
- MyBatis-Plus 使用参数化查询（PreparedStatement）
- Spring Security AuthenticationManager 验证密码 hash
- 即使注入成功查到用户，BCrypt 密码比对也会失败

## 安全架构总结

```
请求流程：
浏览器(HTTPS) → Nginx(SSL终止) → Spring Boot

安全防护层次：
1. 传输层：HTTPS/TLS 加密（Nginx + SSL证书）
2. 认证层：JWT Token（HS256签名 + 黑名单机制）
3. 防重放层：timestamp + nonce（Redis SETNX，5分钟窗口）
4. 授权层：RBAC 角色控制 + 水平越权拦截
5. 审计层：操作日志切面（记录用户、IP、操作、时间）
6. 数据层：MyBatis-Plus 参数化查询防 SQL 注入
```
